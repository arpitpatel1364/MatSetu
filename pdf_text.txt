EXISTING SYSTEM
■ India's current election process relies on Electronic Voting Machines (EVMs) and Voter Verifiable
Paper Audit Trails (VVPATs) that operate entirely offline, making real-time national tallying impossible.
■ Voter authentication depends solely on physical EPIC (Voter ID) cards and manual register checks,
which are slow, error-prone, and susceptible to impersonation.
■ There is no centralised, live view of turnout across booths — District Election Officers receive data
hours after polling closes.
■ Worker deployment and movement cannot be tracked in real-time, making cross-booth fraud detection
impossible.
■ Geographic analysis of voting patterns requires days of manual data aggregation after election results
are declared.
■ There is no cryptographic proof that a specific vote was counted — voters must trust the process
without any technical verifiability.

NEED OF NEW SYSTEM
Real-time biometric voter authentication using face recognition (InsightFace ArcFace) eliminates
■
impersonation entirely.
■ A live, 8-level geographic vote tally (country → state → division → district → taluka → block → village
→ booth) gives every admin tier an instant drill-down view.
■ Worker GPS tracking with anomaly detection flags cross-booth movement, late logins, and suspicious
re-authentication events in real time.
■ PostgreSQL Row-Level Security (RLS) policies enforce geographic data isolation — no admin can see
outside their own scope.
■ A cryptographic hash chain on every vote and a zero-knowledge receipt scheme give India's first
end-to-end verifiable digital election.
■ Federated Qdrant clusters, state-partitioned PostgreSQL, and regional Redis make the system
horizontally scalable to the full 950M voter electorate.

The "Fail-Safe" Protocol
MatSetu does not reject a voter solely because the AI fails. It triggers a Tiered Authentication:
AI Fail: The system flags "Low Confidence Match."
Fallback 1 (Aadhaar OTP): A 6-digit OTP is sent to the voter's linked mobile number.
Fallback 2 (Manual Inspection): If the voter has no phone or an injury, the Constituency Admin
can perform a "Video KYC" override. The booth worker captures a photo of the injury/ID, and a
remote officer approves the vote manually.
Exception Log: All manual overrides are stored in the audit_log with a specific
ANOMALY_OVERRIDE flag for post-election review.

You are right that faces change. However, modern AI like ArcFace and InsightFace (used in our
stack) focus on "Deep Features"—the bone structure, eye-to-ear distance, and orbital sockets
—which remain relatively stable for 5 to 8 years.
Rolling Re-Enrollment: MatSetu uses a "Success-Based Update" logic.
If a voter’s face matches with a high confidence score (e.g., $0.92$), the system automatically
updates their template with the new photo.
This ensures the database "ages" with the voter, making a manual 2-year update unnecessary.
Aadhaar Integration: By syncing with the UIDAI (Aadhaar) database, MatSetu can pull updated
photos from recent SIM card registrations or bank KYC updates, keeping the "Golden Image"
fresh.

❖BOOTH WORKER (Booth Operator)

➤VOTER SCAN
Scans voter's EPIC or E-EPIC QR code using the booth webcam. OCR extracts voter ID, name, DOB.
➤ FACE AUTHENTICATION
Verifies voter's live face against Qdrant enrollment vector. Liveness check prevents photo spoofing.
➤ OTP DISPATCH
After face match, triggers OTP SMS to voter's registered mobile. Worker cannot see or intercept the OTP.
➤ VOTE SUBMISSION
After OTP confirmation, submits the vote to the hub server. Hash chain entry + 8 Redis INCR operations fire
atomically.
➤ RE-AUTHENTICATION
Worker must re-authenticate with face every 30 minutes during shift. Drift in similarity score triggers alert.

FUTURE ENHANCEMENTS
➤ AI Anomaly Heatmap
D3.js choropleth heatmap where colour intensity represents anomaly density per village, pre-computed every
5 minutes by Celery. Allows District Admin to identify concentrated suspicious activity at a glance.
➤ Turnout Prediction Engine
ML model trained on ECI historical turnout data (2019 + 2024 Lok Sabha) predicts expected hourly turnout
per constituency. Alert fires when actual turnout deviates >15% from prediction for 30+ minutes — a
suppression signal.
➤ Cross-Booth Impossible Movement Detection
Redis sorted set tracks worker logins by timestamp. Celery task checks travel velocity: if a worker appears at
two booths >10 km apart within 10 minutes, an impossible movement alert fires immediately.
➤ Offline Mesh Networking (LoRa / Wi-Fi Direct)
For Arunachal Pradesh, Ladakh, Andaman & Nicobar, and Lakshadweep — booths with no cellular
connectivity relay encrypted votes through the nearest connected booth via mDNS broadcast + mTLS relay.
➤ Voter-Verified ZK Proof (Helios-style)
Post-election, voters can verify their specific vote was counted by supplying their random salt. The published
ZK commitment set proves inclusion without revealing candidate selection. Based on Ben Adida's Helios
Voting scheme.
➤ Predictive Booth Failure Monitoring
booth_agent.py heartbeat (every 30 sec) includes battery %, network signal, camera FPS, OCR latency. If
battery <20% AND network <-90 dBm AND SQLite queue >10, a technician dispatch alert fires before the
booth goes offline.
➤ ECI ERMS Integration
Nightly differential sync with Election Commission's Electoral Roll Management System (ERMS). New voter
registrations, deletions, and address changes propagate automatically to voters table and Qdrant enrollment
queue.
➤ Digital VVPAT
A QR receipt that the voter scans with their own phone shows their candidate selection on their screen for 5
seconds before the vote is sealed. No paper required. Satisfies VVPAT legal requirements digitally.
