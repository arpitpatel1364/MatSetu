1. Executive Summary — v2.0 Expansion
MatSetu v2.0 expands the original Gujarat-focused platform to all 28 Indian states and 8 Union Territories. It introduces a 7-tier administrative hierarchy that mirrors India's actual governance structure — from Election Commission level all the way down to individual polling booths. Every vote is now tagged with the full geographic chain: country → state → division → district → taluka → block → village → booth.
Three major new capabilities are added in this version:
Area-wise vote tracking: every vote_ledger row carries the full 8-level location chain. Admins at any level see only their own area's data, enforced by PostgreSQL row-level security.
Worker location intelligence: every booth worker login captures GPS coordinates, device fingerprint, and face embedding. The system detects cross-booth worker movement, unexpected login locations, and shift violations.
District-level administration: new District Admin and Division Admin roles give local election officers granular visibility into their own territory without access to other districts.

7. Research & Innovation Ideas
The following are researched innovations that would significantly enhance MatSetu beyond its core voting function. Each idea is grounded in real technology and feasibility for the Indian election context.

7.1 AI Anomaly Heatmap — Geographic Fraud Visualization
Instead of a simple alert list, District Admins see a live choropleth heatmap of their district where colour intensity represents anomaly density. The AI anomaly scoring system aggregates: failed liveness attempts, duplicate scan attempts, GPS violations, and abnormal vote rates — normalized per booth and rendered as a village-level heatmap.
Implementation: D3.js choropleth on the admin dashboard consuming GET /api/v1/admin/district/{id}/anomaly-heatmap.
Data: Pre-computed per village in Redis every 5 minutes by Celery beat task. Instant render, no complex query.
Value: Allows the District Admin to see at a glance which sub-areas of the district have concentrated suspicious activity — without reading through thousands of individual alerts.
Research basis: Similar geospatial anomaly visualization is used by the ECI's existing GIS systems for monitoring polling station accessibility.

7.2 Turnout Prediction Engine
A ML model trained on historical ECI turnout data (available publicly for 2019 and 2024 general elections) predicts expected hourly turnout curves per constituency. The live dashboard shows actual vs predicted turnout with a deviation flag when actual falls more than 15% below prediction — a potential suppression signal.
Model: Simple time-series regression per constituency using historical election data, weather API, and day-of-week patterns.
Input features: constituency type, rural/urban ratio, 2019/2024 actual turnout, time of day, weather (OpenWeatherMap API).
Output: Predicted votes/hour curve. Real-time deviation score. Alert if deviation >15% for 30+ minutes.
Value: Catches suppression events, booth malfunctions, and infrastructure failures early — while there is still time to intervene.

7.3 Cross-Booth Worker Movement Detection Network
When a worker logs into a booth, their GPS coordinates are broadcast to a lightweight movement detection service that tracks all workers across the district simultaneously. If any worker appears at two booths within a physically impossible travel time (based on driving distance via Google Maps API), an alert fires.
Implementation: Redis sorted set keyed by worker_id, scored by login timestamp. Celery task checks velocity = distance / time_delta for each new login.
Alert threshold: If worker appears to travel >10km in <10 minutes, flag as impossible movement.
Value: Detects booth stuffing operations where one person authenticates at multiple locations using stolen credentials or face spoofing that passed liveness.

7.4 Offline Resilience with Mesh Networking
In remote areas of states like Arunachal Pradesh, Ladakh, and island territories (Andaman & Nicobar, Lakshadweep), there may be no cellular connectivity at all. The proposal is a mesh network between booths in the same village using Wi-Fi Direct or LoRa radio, so votes from offline booths can be relayed through connected booths.
Architecture: booth_agent.py adds a mesh relay mode. If hub is unreachable, attempt to relay through the nearest active booth detected via mDNS broadcast.
Security: Relayed votes are encrypted with the destination booth's mTLS key. They are stored in the relay booth's SQLite queue and forwarded to the hub with the originating booth's signature intact.
Vote integrity: Hash chain is computed at the originating booth before relay. The hub validates originating booth signature before accepting relayed votes.
Research basis: Similar mesh relay architectures are used in disaster-resilient voting systems in Indonesia and Papua New Guinea.

7.5 Voter-Verified Anonymous Receipt with ZK Proof
Currently, the receipt page shows only "Vote recorded" — it cannot reveal the candidate without compromising secrecy. A zero-knowledge proof system would allow a voter to verify that their specific vote was included in the final count, for their chosen candidate, without revealing which candidate they voted for to any third party.
Scheme: At vote submission, generate a ZK commitment: hash(vote_hash + voter_random_salt). Publish commitments publicly. Voter retains their salt. Post-election, the voter can prove their commitment is in the published set without revealing vote_hash.
Implementation: zk-SNARK using snarkjs library. Proof generation runs client-side in the voter's browser on the receipt page.
Value: Provides cryptographic end-to-end verifiability — the gold standard in academic e-voting research — without compromising anonymity.
Research basis: Based on Helios Voting and the work of Ben Adida (Harvard SEAS) on universally verifiable elections.

7.6 Real-Time Booth Health Monitoring with Predictive Failure
Every booth_agent.py sends a heartbeat every 30 seconds containing: battery level, camera status, network signal strength, OCR engine status, and face recognition model load time. A predictive model flags booths likely to fail before they actually go offline.
Metrics collected: battery_pct, network_rssi_dbm, camera_fps, face_recog_latency_ms, ocr_latency_ms, sqlite_queue_depth, last_successful_hub_sync.
Failure prediction: If battery <20% AND network <-90dBm AND queue_depth >10, flag booth as "high failure risk" and dispatch technician alert.
Dashboard: Constituency Admin sees a booth health grid — green/amber/red — updated every 30 seconds via SSE.
Value: Proactive response to booth failures rather than reactive. Critical in remote constituencies.

7.7 Post-Election Forensic Timeline Reconstruction
After election close, the system can reconstruct a complete second-by-second timeline of every event at every booth — from first voter scan to last vote submitted — using the audit_log and worker_location_trail tables. This is presented as a scrollable visual timeline in the Master Admin dashboard.
Data source: audit_log (every action with timestamp) + worker_location_trail (every GPS event) + vote_ledger (every vote submission).
Timeline events: worker_login, voter_scan, liveness_check, ocr_verify, otp_sent, otp_verified, vote_cast, worker_reauth, anomaly_flagged, worker_logout.
Export: Full timeline exported as a cryptographically signed JSON file alongside the hash chain verification report. Usable as legal evidence.
Value: Provides full forensic reconstruction capability for election dispute resolution — something no current EVM system offers.

7.8 Multi-Language Booth Interface
India has 22 scheduled languages and hundreds of dialects. The booth HTML/JS interface supports dynamic language switching without page reload, covering all 22 official languages using i18next.js. The candidate list, instructions, OTP prompts, and receipt text are all translatable.
Implementation: i18next.js with JSON translation files per language. Language detected from voter's registered state and constituency, with override option for migrant workers.
Indic script support: All text rendered in Unicode. Fonts served locally from booth device (no CDN dependency for core UI).
OTP SMS: Twilio template messages sent in voter's registered language using MSG91 regional language templates.
Value: Critical for voter accessibility in non-Hindi states. Required for constitutional compliance under Article 343-351.

7.9 Election Commission Integration — ERMS & EPIC Sync
The Election Commission already operates the Electoral Roll Management System (ERMS) and Electronic Electoral Photo Identity Card (E-EPIC) systems. MatSetu v2.0 proposes a secure API integration so voter enrollment can be pulled directly from ERMS rather than requiring a separate batch upload.
ERMS sync: Nightly differential sync via secure ECI API. New registrations, deletions, and address changes propagate to voters table and Qdrant enrollment queue automatically.
E-EPIC verification: The document OCR step can cross-verify the QR code on digital E-EPIC cards (which contain voter ID, name, DOB, constituency) instead of relying solely on Tesseract text extraction. Faster and more accurate.
VVPAT analog: MatSetu can generate a digital VVPAT equivalent — a QR receipt that the voter scans, which shows their candidate selection on the voter's own phone screen for 5 seconds before the vote is sealed. No paper required.
Research basis: ECI already provides electoral roll data APIs to authorized parties under the Model Code of Conduct framework.

7.10 Federated Qdrant — Distributed Voter Enrollment at State Level
At 950M voters, a single Qdrant cluster storing all face vectors would require ~1.9TB of RAM (512-D FP32 × 950M vectors). Instead, the system uses a federated Qdrant architecture where each state runs its own Qdrant cluster sized for its voter count.
Architecture: State-level Qdrant clusters (e.g., UP-Qdrant for 150M UP voters, MH-Qdrant for 100M Maharashtra voters). The Central Hub routes voter face verification requests to the correct state cluster based on the voter's registered state.
Routing: GET /api/v1/voter/verify-face detects which state Qdrant to query from the booth's location_chain. A voter from Gujarat attempting to vote at a booth in Maharashtra is automatically rejected.
Cost: State-level clusters are proportionally sized. UP (150M voters) needs ~300GB RAM; Goa (1.2M voters) needs only ~2.4GB. Right-sized per state.
Consistency: Pre-election enrollment sync ensures every voter's face vector exists in exactly one state cluster. No cross-cluster search required during voting.


8. Updated Scale Estimates — All India

8.1 Infrastructure Changes for All-India Scale
Qdrant: Federated per state. Each state cluster independently sized. No single cluster exceeds 300GB.

1. About the Project

MatSetu (meaning 'vote bridge') is a next-generation, AI-powered Digital Election Management Platform designed to serve the entire Indian electorate. Version 2.0 supersedes the Gujarat-only v1.0 and scales to all 28 states and 8 Union Territories, covering approximately 950 million registered voters across 1 million+ polling booths.

The platform replaces paper-based voter rolls and manual tallying with a real-time, biometric-authenticated, cryptographically-secured voting pipeline.

2. Existing System (Problems)

India's current election process relies on Electronic Voting Machines (EVMs) and Voter Verifiable Paper Audit Trails (VVPATs) that operate entirely offline, making real-time national tallying impossible.
Voter authentication depends solely on physical EPIC (Voter ID) cards and manual register checks — slow, error-prone, and susceptible to impersonation.
There is no centralised, live view of turnout across booths. District Election Officers receive data hours after polling closes.
Worker deployment and movement cannot be tracked in real-time, making cross-booth fraud detection impossible.
Geographic analysis of voting patterns requires days of manual data aggregation after results are declared.
There is no cryptographic proof that a specific vote was counted — voters must trust the process without any technical verifiability.


3. Need for the New System

Real-time biometric voter authentication using face recognition (InsightFace ArcFace) eliminates impersonation entirely.
A live 8-level geographic vote tally (country → state → division → district → taluka → block → village → booth) gives every admin tier an instant drill-down view.
Worker GPS tracking with anomaly detection flags cross-booth movement, late logins, and suspicious re-authentication events in real time.
PostgreSQL Row-Level Security (RLS) policies enforce geographic data isolation — no admin can see outside their own scope.
A cryptographic hash chain on every vote and a zero-knowledge receipt scheme give India's first end-to-end verifiable digital election.
Federated Qdrant clusters, state-partitioned PostgreSQL, and regional Redis make the system horizontally scalable to the full 950M voter electorate.

6. Challenges & Solutions


The Fail-Safe Protocol (Tiered Authentication)
AI face match fails → system flags 'Low Confidence Match'.
Fallback 1 — Aadhaar OTP: 6-digit OTP sent to voter's linked mobile number.
Fallback 2 — Booth OTP Print: if SMS delivery fails (no signal), thermal slip printed and handed to voter face-down. 60-second expiry.
Fallback 3 — Manual Inspection (Video KYC): if voter has no phone or injury, Constituency Admin performs remote override. Booth worker captures photo of injury/ID; remote officer approves manually.
All manual overrides are stored in audit_log with ANOMALY_OVERRIDE flag for post-election review.

Face Aging — Rolling Re-Enrollment
ArcFace and InsightFace focus on deep structural features (bone structure, eye-to-ear distance, orbital sockets) which remain stable for 5–8 years. MatSetu uses Success-Based Update logic:
If a voter's face matches with high confidence (≥ 0.92), the system automatically updates their template with the new photo.
This ensures the database 'ages' with the voter, eliminating the need for a manual 2-year update cycle.
Aadhaar Integration: by syncing with the UIDAI (Aadhaar) database, MatSetu pulls updated photos from recent SIM card registrations or bank KYC, keeping the Golden Image fresh.

1. Executive Summary
MatSetu (मतसेतु) is a hardened digital election management platform built as a direct evolution of the Senterix-AI surveillance stack. It repurposes and extends the InsightFace + Qdrant + FastAPI core for electoral authentication — replacing physical ballot boxes with a cryptographically sealed, face-verified digital vote.
Election corruption is eliminated at two layers: voter authentication (impersonation, fake votes) and vote counting (manipulation after submission). Every vote is gated behind 3-factor authentication. Every counted vote is sealed with a SHA-256 hash chain enforced at the PostgreSQL permission level.

Core Design Principles
Zero Trust — every request requires cryptographic proof of identity
Face as Identity — ArcFace 512-D embeddings are the primary identity anchor
Append-Only Ledger — votes are written once with SHA-256 hash chains, impossible to edit without detection
Role Isolation — Workers, Gujarat Admins, and Master Admin operate in completely separated permission scopes
Offline-First Booths — Worker nodes queue votes locally in SQLite if internet drops, sync on reconnect
Live Transparency — Master Admin sees vote counts update in real-time via SSE

2. System Roles & Access Model
Four completely isolated roles exist at the database, API, and UI layer.


Authentication Per Role
Voter: Face recognition (ArcFace 512-D via Qdrant) + OTP (SMS via Celery) + Document OCR match
Worker: Face recognition (InsightFace pipeline) + Device PIN + 4-hour JWT

3. System Architecture

4. Database Design
PostgreSQL 16 is the primary datastore. The schema is designed for ACID compliance, auditability at scale, and tamper detection through SHA-256 hash chains. All tables are named in snake_case. The vote_ledger is INSERT-only — UPDATE and DELETE are revoked at the PostgreSQL role level.

4.2 Table: voters
Stores every registered Gujarat voter. Aadhaar and phone numbers are never stored in plaintext — only their SHA-256 hashes. The face_vector_id links this record to the corresponding Qdrant point in the gujarat_voter_roll collection.


4.3 Table: vote_ledger  (APPEND-ONLY — most critical table)
The heart of the system. Each row represents one submitted vote. The PostgreSQL role vote_writer has INSERT and SELECT only — UPDATE and DELETE are explicitly revoked. Election corruption happens at two layers: voter authentication (impersonation, fake votes) and vote counting (manipulation after submission). MatSetu (मतसेतु) eliminates both. 4.2 Voter Authentication Flow (Step by Step)
Worker presses 'Scan Voter' in the booth HTML UI.
Webcam frame captured by Python cv2 in booth_agent.py.
RetinaFace detects face bounding box. If no face: prompt retry (max 3 attempts).
MiniFASNet liveness check: is this a real face? If spoof detected: hard reject, log to audit, store face crop in MinIO.
ArcFace generates 512-D embedding from the aligned and cropped face.
booth_agent.py sends POST /api/voter/verify-face with the embedding vector (mTLS + booth JWT).
FastAPI hub queries Qdrant collection 'gujarat_voter_roll'. Nearest neighbour search, cosine similarity.
If similarity > 0.72: voter identity confirmed. Hub returns {voter_id, name, constituency, already_voted}.
If already_voted is true: hard reject. Log attempt. Alert Gujarat Admin via SSE.
Worker receives voter info. Scans voter's Aadhaar/Voter ID via webcam. Tesseract extracts name and DOB.
FastAPI compares OCR result against voter record. Both must match to proceed.
Hub triggers OTP via Celery task: sends SMS to voter's registered phone via MSG91/Twilio.
Voter enters OTP on booth screen. Hub validates. Issues voter_session_token (UUID, 5-minute TTL, single-use).
Worker's HTML screen shows candidate list for voter's constituency. Voter selects candidate.
Worker clicks Submit. POST /api/vote/submit with {candidate_id, voter_session_token, worker_JWT}.
Hub validates all tokens, checks idempotency key, writes to vote_ledger with hash chain. Returns receipt_code.
Booth UI shows QR code receipt to voter. Redis INCR updates live tally. SSE pushes update to admin dashboards.

4.3 Worker Authentication Flow
Workers authenticate themselves before they can operate any booth. This prevents unauthorized terminal use.
Worker sits at booth laptop. HTML page shows: 'Worker Login — Please look at camera'.
Same InsightFace pipeline: RetinaFace → MiniFASNet liveness → ArcFace 512-D embedding.
POST /api/auth/worker-login — hub searches Qdrant 'worker_roster' collection.
If match found: issue 4-hour JWT. Store in OS keychain on booth device (not in browser localStorage).
If no match or liveness fail: log attempt, lock terminal for 30 seconds, alert Gujarat Admin.
Every API call from booth includes JWT in Authorization: Bearer header.
When JWT expires: worker must re-authenticate by face. This is intentional — no persistent sessions.

NOTE: MiniFASNet liveness detection catches printed photos and screen replay attacks. For maximum security at high-priority booths, add an active liveness blink challenge as a second layer. DeepFace supports this natively.
4.4 Face Enrollment (Pre-Election Batch Process)
Before election day, all 50M Gujarat voters' faces must be enrolled into Qdrant. This is a Python batch script.
Source data: Voter registration photos from Election Commission database (JPEG/PNG per voter).
Script: enroll_voters.py — reads photo, runs ArcFace, upserts 512-D vector into Qdrant 'gujarat_voter_roll'.
Payload per point: {id: voter_id, vector: [512 floats], payload: {name, constituency_id, enrolled_at}}.
Workers enrolled separately: enroll_workers.py → Qdrant 'worker_roster' collection.

CREATE TABLE vote_ledger (
  vote_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  voter_id        UUID NOT NULL REFERENCES voters(voter_id) UNIQUE,  -- one vote enforced
  candidate_id    UUID NOT NULL REFERENCES candidates(id),
  constituency_id UUID NOT NULL REFERENCES constituencies(id),
  booth_id        UUID NOT NULL REFERENCES booths(id),
  worker_id       UUID NOT NULL REFERENCES workers(id),
  submitted_at    TIMESTAMPTZ DEFAULT NOW(),
  vote_hash       TEXT NOT NULL,  -- SHA-256(voter_id||candidate_id||timestamp||booth_secret)
  prev_hash       TEXT,           -- vote_hash of the previous INSERT (hash chain)
  idempotency_key TEXT UNIQUE NOT NULL,  -- SHA-256(voter_id||booth_id||election_id)
  is_valid        BOOLEAN DEFAULT TRUE
);
-- CRITICAL: Revoke UPDATE and DELETE from all app users
REVOKE UPDATE, DELETE ON vote_ledger FROM vote_writer;
GRANT INSERT, SELECT ON vote_ledger TO vote_writer;

8.2 Hash Chain — How Vote Integrity Works
Each row in vote_ledger stores a hash of the previous row. This makes tampering detectable — if any vote is modified, all subsequent hashes become invalid.

# In vote_service.py
import hashlib

def compute_vote_hash(voter_id, candidate_id, timestamp, booth_secret):
    raw = f'{voter_id}{candidate_id}{timestamp}{booth_secret}'
    return hashlib.sha256(raw.encode()).hexdigest()

def get_prev_hash(db_session):
    last_vote = db_session.exec(
        select(VoteLedger).order_by(VoteLedger.submitted_at.desc()).limit(1)
    ).first()
    return last_vote.vote_hash if last_vote else 'GENESIS'

# verify_hash_chain.py (run post-election for audit)
def verify_chain(db_session):
    votes = db_session.exec(select(VoteLedger).order_by(VoteLedger.submitted_at)).all()
    for i, vote in enumerate(votes[1:], 1):
        expected_prev = votes[i-1].vote_hash
        if vote.prev_hash != expected_prev:
            print(f'CHAIN BROKEN at vote {vote.vote_id}')
            return False
    print('Chain verified: all votes intact')
    return True

9. Server-Side Scale Design
Gujarat has 50M+ registered voters, 182 constituencies, and ~50,000 polling booths. Peak election day load is the hardest engineering problem. Here is how the system handles it without data loss, corruption, or downtime.

9.1 Load Estimates

9.2 FastAPI Scaling
Deploy 8–16 FastAPI pods behind NGINX with least_conn load balancing.
Each pod: uvicorn --workers 4 --worker-class uvicorn.workers.UvicornWorker. Total: 64 async coroutine pools.
FastAPI is fully async (async def routes). No blocking I/O — all DB and Redis calls use async drivers.
Stateless design: zero session state in-process. All state in Redis or PostgreSQL.
Kubernetes HPA: auto-scale pods when CPU > 60% or request queue > 500ms p99.

9.3 PostgreSQL Scaling
Primary: 1 PostgreSQL 16 node (32-core, 256GB RAM) for all writes.
Read replicas: 3 replicas for read-heavy admin dashboards and audit log queries.
PgBouncer in transaction mode: pools 50,000 potential connections down to ~200 actual DB connections.
vote_ledger: append-only with no UPDATE/DELETE locks — pure INSERT throughput, no row-level contention.
audit_log: partitioned by month (PARTITION BY RANGE logged_at) — fast inserts, independent partition maintenance.
Indexes: vote_ledger(voter_id) UNIQUE, vote_ledger(booth_id, submitted_at), audit_log(actor_id, logged_at).

9.4 Redis Cluster
Redis Cluster: 6 nodes (3 primary + 3 replica) for HA. Sharded by hash slot.
Live tally: INCR constituency:{id}:votes on every vote submission. O(1) operation. No SELECT COUNT(*) needed.
SSE feed: PUBLISH vote_events {constituency_id, total_votes} on every vote. Admin EventSource listeners receive instantly.
JWT blacklist: SET jti:{token_id} EX 3600 — invalidate tokens on worker logout or security event.
Rate limit counters: INCR rate:{ip} EX 60 — 100 requests/minute per booth IP, enforced at FastAPI middleware level.
OTP tokens: SET otp:{voter_id} {hashed_otp} EX 300 — 5-minute TTL, auto-expires.

9.5 Qdrant Scaling
3-node Qdrant cluster: distributed sharding of 50M voter vectors across nodes.
HNSW index: m=16, ef_construction=100. Approximate nearest-neighbour at 50M scale in ~15ms.
Two collections: 'gujarat_voter_roll' (50M vectors), 'worker_roster' (~50,000 vectors).
Qdrant gRPC API used from FastAPI for lower serialization overhead at high request rates.
Cache recently verified voters in Redis: SET verified_voter:{voter_id} 1 EX 60. Avoids repeat Qdrant calls if worker re-scans.

9.6 Celery Async Workers
16 Celery worker processes. Redis as message broker. Priority queues: OTP (high), PDF receipt (low).
OTP task: send_otp.delay(voter_id, phone_hash) — Twilio HTTP call in background, never blocks the vote flow.
Receipt PDF: generate_receipt.apply_async(vote_id, countdown=5) — deferred 5 seconds after vote confirmed.
Anomaly scanner: Celery beat task runs every 60 seconds — checks for booths with abnormal vote rates or repeated liveness failures.

9.7 Offline Booth Resilience
Rural Gujarat booths may have intermittent or no internet. The booth_agent.py handles this gracefully.
Every submitted vote is first written to a local SQLite queue in local_queue.py before sending to hub.
If hub is unreachable: vote is stored locally. Booth UI shows 'Queued (offline) — will sync on reconnect'.
On reconnect: booth_agent.py replays the queue to hub in order, with idempotency_key preventing double-counting.
Hub rejects duplicates: 409 Conflict if idempotency_key already exists in vote_ledger.
voter_session_token has 5-minute TTL — voter cannot leave and come back to submit an already-verified vote.

10. Security & Anti-Corruption Model
The most important question about any voting system: how do we know the results are real and untampered? Here is the complete answer for MatSetu (मतसेतु).

10.1 Attack Vectors and Mitigations

10.2 Why No Single Person Can Corrupt the Results
Master Admin: Can read everything but cannot write to vote_ledger. PostgreSQL permission is SELECT-only for master_reader role.
Gujarat Admin: Read-only role at DB level. Cannot touch any voting table.
Workers: Can only INSERT into vote_ledger via /api/vote/submit. They get back only a receipt_code. They cannot read votes.
DBA: Even the database administrator cannot silently modify vote_ledger without breaking the hash chain.
Senterix platform owner: Code changes would require redeployment — logged by Kubernetes and Git. Runtime tampering is impossible.

10.3 Independent Post-Election Audit
After election close, Master Admin exports the full vote_ledger as a CSV with all vote_hash and prev_hash values.
The Election Commission runs verify_hash_chain.py against the export independently.
The script recomputes every vote_hash from raw fields + booth_secret and confirms prev_hash chain is unbroken.
If even one row has been modified: the chain breaks at that point and is immediately visible.
Audit log export covers every single action taken by every actor, with timestamps and IP addresses.

10.4 Voter Privacy
Vote secrecy: vote_ledger stores voter_id and candidate_id separately. No API joins them in a public response.
Receipt page only shows: 'Your vote was recorded' — never reveals candidate.
Aadhaar numbers never stored in plaintext — only SHA-256(aadhaar). Cannot be reversed.
Phone numbers hashed. Only OTP service maps hash to actual number for SMS dispatch.
Face crop audit photos (stored in MinIO) are encrypted at rest and accessible only to the audit_service role.
PostgreSQL row-level security: Gujarat Admin can only SELECT rows for their own region.
